# Linux コマンド研修アプリ 基本設計書

## 1. 文書概要

### 1-1. 文書名
Linux コマンド研修アプリ 基本設計書

### 1-2. 目的
本書は、Linux コマンド研修アプリのシステム構成、機能構成、データ構成、外部連携、運用方針などの基本設計を定義することを目的とする。  
要件定義で整理した内容をもとに、開発・実装・テスト・運用へ進めるための設計方針を明確化する。

### 1-3. 対象システム
- Linux コマンド研修アプリ
- ブラウザ上で Linux コマンドを実行できる研修用 Web アプリケーション

### 1-4. 前提
- 同時接続数は最大30人程度を想定する
- 受講者ごとに分離された Linux 実行環境を提供する
- 実行環境は Docker コンテナを利用する
- ターミナル UI はブラウザ埋め込み方式とする
- 今後の機能追加や UI 改修に対応できる構成とする

---

## 2. システム全体構成

### 2-1. システム構成概要
本システムは、以下のコンポーネントで構成する。

- フロントエンド
- バックエンド API
- WebSocket 通信処理
- セッション管理基盤
- Linux 実行環境管理基盤
- 管理用機能
- ログ・監視基盤

### 2-2. 想定技術構成
- フロントエンド: Next.js
- UI 実装: React
- ターミナル UI: xterm.js
- バックエンド: Node.js
- API 実装: Express または Fastify
- リアルタイム通信: WebSocket
- セッション管理: Redis
- Linux 実行環境: Docker
- リバースプロキシ: Nginx
- インフラ: AWS EC2 などのクラウド基盤

### 2-3. 構成イメージ
1. 受講者がブラウザからアプリへアクセスする  
2. フロントエンドが API 経由で認証・ユーザー情報を取得する  
3. ターミナル画面表示時に WebSocket 接続を確立する  
4. バックエンドが対象受講者の Docker コンテナへ接続する  
5. コマンド入力内容をコンテナ内シェルへ送信する  
6. 実行結果をリアルタイムでブラウザへ返却する  
7. 管理者は管理画面から受講者状態やコンテナ状態を確認する

## 3. アーキテクチャ設計

### 3-1. 採用アーキテクチャ
本システムは以下の構成を採用する。

- フロントエンド分離型 Web アプリケーション
- API サーバと WebSocket サーバの論理分離
- 受講者単位のコンテナ実行環境
- セッション情報の外部ストア管理
- 将来的なスケールアウトを見据えた疎結合構成

### 3-2. 初期構成
初期リリースでは、運用簡易性を重視し、以下の単一構成を想定する。

- 1台のアプリケーションサーバ
- Next.js
- Node.js API / WebSocket
- Docker ホスト
- Redis
- Nginx

### 3-3. 将来拡張構成
将来的には以下の拡張を可能とする。

- アプリケーションサーバの複数台構成
- Docker 実行ホストの分離
- Redis の専用化
- ロードバランサ配下での分散
- ECS / Kubernetes 等への移行

## 5. 画面・処理方式設計

### 5-1. フロントエンドの役割
- 認証画面の提供
- 受講者画面の描画
- ターミナル UI の描画
- API 呼び出し
- WebSocket 通信の確立・維持
- 管理画面の描画

### 5-2. バックエンドの役割
- ログイン認証
- ユーザー情報取得
- 受講者環境情報取得
- コンテナ生成 / 停止 / 削除
- WebSocket 中継
- 実行履歴記録
- 管理者向け情報集約

### 5-3. ターミナル処理方式
- ブラウザ上では xterm.js によりターミナルを表示する
- 入力内容は WebSocket でバックエンドへ送信する
- バックエンドは対象コンテナのシェルへ接続し、コマンドを実行する
- 出力内容は逐次ブラウザへ返却する
- ブラウザではリアルタイムに実行結果を表示する

## 6. コンテナ実行環境設計

### 6-1. 基本方針
- 1ユーザーにつき 1コンテナを割り当てる
- 受講者同士の環境を分離する
- ホスト環境に直接アクセスさせない
- 非特権モードで実行する
- 必要最低限の Linux コマンドのみ利用可能にする

### 6-2. コンテナのライフサイクル
- ログイン時またはターミナル接続時に割り当て確認を行う
- 未割当の場合は新規作成する
- 利用中は起動状態を維持する
- 一定時間無操作の場合は停止候補とする
- リセット時は削除後に再生成する

### 6-3. 初期設定
- 受講者用ユーザーを作成する
- ホームディレクトリを用意する
- 課題ファイルを配置する
- 初期ディレクトリ構造を整備する
- 操作ログ取得設定を有効化する

### 6-4. リソース制限
- CPU 制限を設定する
- メモリ制限を設定する
- ディスク容量制限を設定する
- プロセス数上限を設定する
- 必要に応じてタイムアウト制御を行う

### 6-5. セキュリティ制御
- privileged モードは禁止
- root 実行は禁止または最小限に制御
- sudo 利用は禁止
- 不要なネットワーク通信は禁止
- ホストボリュームの過剰マウントは禁止
- 危険コマンドは制御対象とする

## 7. セッション管理設計

### 7-1. セッション管理方針
- ログインセッションは Redis で管理する
- WebSocket セッションとアプリセッションを関連付ける
- 受講者とコンテナ ID の紐付けを保持する
- 同一アカウントの多重接続制御を行う

### 7-2. 保存対象
- ユーザーID
- ログイン状態
- ロール情報
- 接続中コンテナ ID
- WebSocket 接続状態
- 最終操作時刻
- 接続開始時刻

### 7-3. セッションタイムアウト
- 一定時間無操作時に警告表示を行う
- 一定時間経過後は自動切断または停止候補とする
- 管理者操作で強制切断可能とする

## 9. API 設計方針

### 9-1. API 基本方針
- REST API を基本とする
- 認証が必要な API はトークンまたはセッションチェックを行う
- フロントエンドからの呼び出しを前提とする
- 管理者専用 API はロールチェックを行う

### 9-2. API 例

#### 認証系
- POST /api/auth/login
- POST /api/auth/logout
- GET /api/auth/me

#### 受講者系
- GET /api/course/current
- GET /api/task/list
- GET /api/environment/status
- POST /api/environment/reset

#### 管理者系
- GET /api/admin/users
- GET /api/admin/containers
- GET /api/admin/logs
- POST /api/admin/environment/reset
- POST /api/admin/environment/stop

### 9-3. WebSocket 用途
- ターミナル入力送信
- ターミナル出力受信
- 接続状態通知
- セッション切断通知
- 環境状態変更通知

## 13. セキュリティ設計

### 13-1. 通信
- HTTPS を必須とする
- WebSocket も WSS を前提とする
- 認証済みユーザーのみ利用可能とする

### 13-2. アプリケーション
- 権限チェックを実施する
- 入力値バリデーションを実施する
- ログイン試行制御を検討する
- CSRF / XSS / セッション固定化対策を実施する

### 13-3. 実行環境
- コンテナを分離する
- ホストへの影響を遮断する
- リソース上限を設ける
- 不要なコマンドや機能を制限する

## 15. 性能設計

### 15-1. 前提
- 同時接続数 30人
- 1ユーザー1コンテナ
- 基本的な Linux コマンド演習が中心
- 重いビルド処理や大規模インストールは想定しない

### 15-2. 設計方針
- 事前起動済みコンテナを一定数用意する
- コンテナごとに CPU / メモリ制限を設定する
- セッションを外部管理し、再接続に備える
- API と WebSocket の責務を整理する
- 将来の水平分割に耐える構成とする

## 18. UI 改修に関する設計方針

### 18-1. 基本方針
本システムの UI は、研修内容や運用要件の変更に応じて、今後改修される可能性がある。  
そのため、基本設計時点では UI を固定的なものとせず、フロントエンドとバックエンドを疎結合に保ち、UI 改修の影響が業務ロジックへ波及しにくい構成を採用する。

### 18-2. 設計上の考慮事項
- 画面レイアウト変更に耐えられるコンポーネント設計とする
- API は画面実装に依存しすぎない構造とする
- ターミナル領域、課題表示領域、操作ボタン群は独立部品化する
- 将来的なスマホ / タブレット最適化やデザイン変更を考慮する
- 管理画面も同様に画面再設計可能な構造とする

---

## 19. まとめ
本システムは、Linux コマンド研修を安全かつ効率的に実施するための Web アプリケーションであり、受講者ごとに分離された Linux 実行環境を提供する。  
基本設計では、同時30人接続を前提に、運用性・安全性・拡張性を担保しつつ、今後の UI 改修にも柔軟に対応できる構成を採用する。
