## 1. HTTPS／WSS 利用の確認

### 1-1. 目的

要件定義「通信は HTTPS 化すること」に対し、本番相当環境で TLS 終端とブラウザ接続を確認する。

### 1-2. 手順（インフラ）

1. `infra/` で証明書を用意し、Nginx で HTTPS を有効化する。手順は **`docs/06_利用手順.md` の「5. HTTPS（TLS）を有効にする場合」** および **`docs/05_インフラ構成書.md`** を参照する。
2. ブラウザで **`https://<ホスト>/`** を開き、接続が暗号化されていることを確認する（開発時は自己署名の警告が出る場合がある）。
3. フロント（例: 3000）・API（例: 3001）を **Nginx 配下の同一オリジン**で公開する場合は、`FRONTEND_ORIGIN` と Cookie の `Secure` 属性（本番 `NODE_ENV=production`）を合わせる。

### 1-3. WebSocket（WSS）

- ブラウザのターミナルは **`NEXT_PUBLIC_WS_URL`** で WebSocket URL を指定する。HTTPS ページからは **`wss://`** を用いる（混在コンテンツを避ける）。
- Nginx で `/ws/` をバックエンドにプロキシし、**Upgrade / Connection** ヘッダを転送する設定にする（`docs/05_インフラ構成書.md` の方針に従う）。

### 1-4. ローカル開発

- `http://127.0.0.1:3000` のみの場合は **TLS なし**でよい（本番前にステージングで HTTPS/WSS を必ず確認する）。

## 2. 権限チェック（管理者 API・画面）の網羅

### 2-1. バックエンド（Fastify）

`hooks.requireAuth` と `hooks.requireRole("admin")` により、**`/api/admin/*`** は **認証済みかつロール `admin`** のみアクセス可能。

| メソッド | パス | 保護 |
|----------|------|------|
| GET | `/api/admin/users` | adminOnly |
| GET | `/api/admin/users/:userId` | adminOnly |
| GET | `/api/admin/containers` | adminOnly |
| GET | `/api/admin/logs` | adminOnly |
| POST | `/api/admin/environment/reset` | adminOnly |
| POST | `/api/admin/environment/stop` | adminOnly |
| POST | `/api/admin/users/:userId/disconnect` | adminOnly |

受講者向けは `requireRole("learner")`（`/api/course/*`、`/api/task/*`、`/api/environment/*` 等）。  
**認証のみ**: `/api/auth/logout`、`/api/auth/me`。

### 2-2. フロントエンド（Next.js）

| 領域 | コンポーネント | 内容 |
|------|----------------|------|
| 管理者ダッシュボード | `app/admin/(dash)/layout.tsx` | `<RequireAuth role="admin">` |
| 受講者アプリ | `app/learner/(app)/layout.tsx` | `<RequireAuth role="learner">` |

画面側のガードは **UX・誤遷移防止**用。**権限の最終判断は常に API** とする。

### 2-3. 確認方法

1. 受講者でログインし、ブラウザで **`/api/admin/users`** を直接叩く（開発者ツール等）→ **403** となること。
2. 管理者でログインし、同 API が **200** となること。

## 3. コンテナ非特権・外部通信制限

実装は **`server/src/services/docker-runtime.ts`** の `createLearnerContainer` に集約。

| 要件（要件定義 8-4 等） | 実装 |
|-------------------------|------|
| 非特権ユーザーで実行 | `User: "1000:1000"` |
| 特権の濫用抑制 | `CapDrop: ["ALL"]`、`SecurityOpt: ["no-new-privileges:true"]` |
| 外部ネットワーク制限 | `HostConfig.NetworkMode: "none"`（コンテナから外向き通信を遮断） |
| リソース上限 | `Memory`、`NanoCpus`、`PidsLimit` |

研修イメージ側で root シェルを許さないことと、`User` 指定を併せて確認する。

## 4. 性能目標の計測

### 4-1. 要件（抜粋）

- 同時接続 **30 人**で安定動作
- ターミナル入力から画面反映まで **通常 1 秒以内**（目標）
- ログイン後、受講者環境接続開始まで **10 秒以内**（目標・事前起動なし）
- 事前起動済みコンテナがある場合 **3 秒以内**（目標）

### 4-2. 計測の考え方

- **同時 30 人**: 負荷ツール（k6・Apache Bench 等）で **ログインまたはセッション付き API** を並列実行するか、30 ブラウザ／シナリオでスモークする。実施環境の CPU・メモリ・Docker を記録する。
- **遅延**: ターミナルで 1 文字入力から xterm に描画されるまでを **画面録画＋タイムスタンプ** またはブラウザのパフォーマンスログで概算する。
- **接続開始時間**: `/api/environment/reset` またはターミナル接続完了イベントまでの経過時間を計測する。

### 4-3. 記録テンプレート（例）

| 日付 | 同時セッション数 | 平均応答（ms） | エラー率 | メモ |
|------|------------------|----------------|----------|------|
| | | | | |

受入は **`docs/00_進捗管理表.md` フェーズ9** で本番相当条件に近い形で実施する。

## 5. ログ要件（保存対象・保持・検索性）

### 5-1. 要件定義との対応（実装ベース）

| 保存対象（要件 12 章の例） | 実装の置き場所 |
|---------------------------|----------------|
| コマンド実行・警告 | `CommandHistory`（DB）、危険コマンドはサーバ側で検知・記録 |
| 接続／切断 | `ConnectionLog`（DB） |
| 管理者操作 | `AdminAuditLog`（DB） |
| アプリ標準ログ | Fastify / Node の **標準出力**（コンテナなら `docker logs` 等で収集） |

### 5-2. 保持・検索

- **DB**: Prisma マイグレーション方針に従い、長期保持が必要なら **バックアップ・アーカイブ**を別途定義する。
- **検索**: 管理画面のログ API（`/api/admin/logs`）および DB 直接参照（運用者のみ）を想定。

### 5-3. ギャップ

- **ログイン成功／失敗の監査ログを DB に永続化**する要件がある場合は、別タスクで `LoginAudit` 相当のテーブル追加や `/api/auth/login` 内の記録を検討する。現状は **401／200 の応答**と **アプリログ**で追跡可能な範囲に留まる。

## 6. 監視要件（対象・通知）

### 6-1. 死活・依存関係

| エンドポイント | 用途 |
|----------------|------|
| `GET /api/health` | プロセス生存（軽量） |
| `GET /api/ready` | **PostgreSQL・Redis** への接続確認（失敗時 **503**） |

監視ツールは **`/api/ready`** を **30〜60 秒間隔**で叩き、非 200 をアラートに連携する想定。

### 6-2. リソース・接続数

- **Docker**: ホスト側で `docker stats`、または Prometheus / cAdvisor 等で **CPU・メモリ**を監視する（`docs/05_インフラ構成書.md` の監視方針）。
- **受講者接続数**: Redis のセッション数・管理 API のオンライン表示を参照（要件「可視化」）。

### 6-3. 通知

- メール・チャット Webhook は **インフラ側**（CloudWatch・アラート等）で **ready 失敗・ディスク・メモリ**を通知する運用を推奨。

## 7. 運用手順（ユーザー登録・研修前準備・リセット・終了後整理）

### 7-1. ユーザー登録（受講者・管理者）

1. DB にユーザーを登録する。開発時は **`npm run db:seed`**（`server/`）で初期アカウントを投入できる。
2. 本番では **パスワードを必ず変更**し、初期シードユーザーを無効化する。

### 7-2. 研修前

1. **`infra`**: PostgreSQL・Redis・（必要なら）Nginx を起動する（`docs/06_利用手順.md`）。
2. **`server`**: `prisma migrate` / `db push`、環境変数（`DATABASE_URL`、`REDIS_URL`、`DOCKER_SOCKET_PATH`、`LEARNER_IMAGE`）を確認する。
3. **研修用イメージ**: `linuxtrainer-learner:latest` をビルド済みにする（`server/README.md`）。
4. **監視**: `/api/ready` が **200** であることを確認する。

### 7-3. 研修中

- 受講者の **環境リセット**は UI または管理者 API から実行。
- 問題発生時は **管理画面のログ**と **サーバ stdout**、必要に応じて **DB** を確認。

### 7-4. 終了後

1. 受講者コンテナの **停止・削除**（管理 API またはホストで Docker 掃除）。
2. **バックアップ**が必要なら DB をダンプ。
3. 次回に向け **シードユーザーの無効化**・**パスワードローテーション**。
