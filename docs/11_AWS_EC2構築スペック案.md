# AWS EC2 構築スペック案

**基準日**: 2026-03-29  
**目的**: 本リポジトリの研修アプリ（`01_要件定義書`・`05_インフラ構成書`・`infra/docker-compose.yml`）を **単一 EC2 上に Docker Compose で載せる**前提の、インスタンスタイプ・ストレージ・運用上の目安をまとめる。  
実測値は負荷試験（`08_結合受入テスト.md`・k6）で調整すること。

---

## 1. 前提（スケール）

| 項目 | 値・出典 |
|------|----------|
| 想定同時利用 | **最大 30 人**（要件定義 9-1） |
| 研修コンテナ | 1 ユーザー 1 コンテナ。余剰含め **35〜40 コンテナ**を扱える構成を想定（`05_インフラ構成書` 6-1） |
| アプリ実装のコンテナ上限 | **メモリ 128MB / CPU 0.5 vCPU 相当 / Pids 256**（`server/src/services/docker-runtime.ts` の `Memory`・`NanoCpus`・`PidsLimit`） |
| 同一ホストに載せるもの | Nginx・PostgreSQL・Redis・Node API（＋任意で Next）、**Docker Engine と研修コンテナ**（`infra/docker-compose.yml` と整合） |

※ 本ドキュメントは **DB・Redis を EC2 上の Compose と同一ホストに置く**構成を想定する。RDS／ElastiCache へ分離する場合は、EC2 からは **アプリ＋Docker 研修コンテナ**分のメモリ・CPU に集中できる。

---

## 2. リソース見積もり（目安）

### 2-1. 研修コンテナ（ピーク時）

- メモリ: 40 コンテナ × **128MB** ≈ **5.1 GiB**（上限値の合算。常時フルは稀だが、設計上はこのオーダーを確保するのが安全）
- CPU: コンテナごとに **0.5 vCPU 上限** → 理論上は **20 vCPU** に相当する負荷が重なる可能性（実際はシェル操作のため平均は低めになりやすい）

※ 旧来の **256MB/コンテナ** 前提（合算約 10.2 GiB）と比べ、**ホストメモリの押さえ方が約半分**になる。同一人数を運用する場合でも **16 GiB 帯のインスタンスでバランスを取りやすい**。

### 2-2. その他プロセス（同一ホスト）

| コンポーネント | メモリ目安 |
|----------------|------------|
| PostgreSQL | 512 MB〜2 GB（データ量・接続数に依存） |
| Redis（Compose 例） | 最大 **128MB**（`infra/docker-compose.yml` の `maxmemory`） |
| Node API（Fastify） | 256 MB〜1 GB 程度 |
| Nginx | 数十 MB |
| Docker デーモン・メタデータ | 数百 MB〜 |
| OS（Amazon Linux 2023 等） | 512 MB〜1 GB |

### 2-3. 要件定義の「初期サーバ想定」

- **4 vCPU 以上・8GB メモリ以上・SSD**（`01_要件定義書` 9-3）  
→ **開発・検証・小規模 PoC** 向けの下限イメージ。  
**30 人＋40 コンテナ上限**を本番に近い条件で同一ホストに載せる場合は、研修コンテナだけで **約 5 GiB** 相当の上限が乗るため、**実運用では 16 GiB 以上**のメモリを推奨（下記 **推奨帯**）。

---

## 3. EC2 インスタンスタイプ案

### 3-1. 用途別の目安

| 区分 | vCPU | メモリ | 想定用途 |
|------|------|--------|----------|
| **最小（検証・PoC）** | 4 | **16 GB** | 128MB/コンテナ前提でコンテナ合算が約 5 GiB のため、DB・API と併せて **16 GiB** で現実的。同時数を抑えた検証向け |
| **推奨（本番に近い 30 人想定）** | **8** | **32 GB** | DB 成長・ピーク・監視の余裕を含めたバランス型 |
| **余裕（ピーク・将来拡張）** | 16 | 64 GB | 同時コンテナ数増・監視エージェント・ログ・バッファ |

**ファミリー**: バランス型 **M 系**（例: `m6i`, `m7i`, `m6a`）を推奨。  
CPU 偏重の演習が続く場合は **C 系**も候補だが、メモリ不足になりやすいため、まずは **メモリ 32GB 帯の M 系**を優先するのがよい。

### 3-2. AWS 上の表記例（参考）

- 推奨帯の一例: **`m6i.2xlarge`**（8 vCPU / 32 GiB）  
- 最小帯の一例: **`m6i.xlarge`**（4 vCPU / 16 GiB）※128MB/コンテナ前提では 30 人フル時も **256MB 時代より余裕が出やすい**

リージョン・アーキテクチャ（x86 / Graviton）・料金に応じて `m7i` / `m6a` 等へ読み替える。

---

## 4. EBS ボリューム

### 4-1. ルートボリューム

- **タイプ**: **gp3**（汎用 SSD）を推奨  
- **サイズ**: 最低 **30〜50 GB**（OS・Docker イメージレイヤ・ログの余裕）。本番では **50〜100 GB** を検討  
- **IOPS / スループット**: 既定 gp3 で足りない場合のみ引き上げ（DB を同一 EBS 上に置く場合は監視）

### 4-2. データ分離（推奨オプション）

- PostgreSQL データ・Docker イメージ・コンテナの可変層でディスクが枯渇しやすいため、  
  **別 EBS ボリューム**を `/var/lib/docker` や DB データディレクトリにマウントする構成を検討する  
- スナップショット取得・拡張の単位を分けやすくなる

### 4-3. バックアップ

- **EBS スナップショット**の定期取得（および RDS 利用時は RDS のバックアップ設定）  
- 保持期間は運用方針に合わせる（`05_インフラ構成書` 8 章チェックリスト）

---

## 5. ネットワーク・セキュリティ（EC2 周り）

| 項目 | 目安 |
|------|------|
| セキュリティグループ | インターネットからは **443**（および必要なら 80→HTTPS リダイレクト）のみ。SSH は **22 を管理者 IP のみ** などに制限 |
| パブリック | Nginx または ALB をフロントに。DB/Redis は **プライベートのみ**（`05` の本番の考え方と同じ） |
| TLS | ACM + ALB、または EC2 上で Let’s Encrypt + Nginx（`docs/06_利用手順.md`・`07`） |

---

## 6. 構成のバリエーション（スケール時）

| パターン | 内容 |
|----------|------|
| **単一 EC2 + Compose** | 本ドキュメントの主対象。コスト・運用がシンプル |
| **RDS + EC2** | DB を切り出し、EC2 はアプリ＋Docker。EC2 メモリをコンテナに振りやすい |
| **ALB + 複数 EC2** | 水平分割が必要になった段階で検討。セッションは Redis やスティッキーで設計 |

## 8. コスト見積もり（リージョン: **us-west-2 オレゴン**）

**前提**

- リージョンは **US West (Oregon) `us-west-2`**。EC2 をオレゴンに置く想定で記載する。  
- **オンデマンド料金**の目安（**税別**）。AWS の公開価格は変更されるため、見積時は [AWS 料金ページ](https://aws.amazon.com/ec2/pricing/on-demand/) または **料金計算ツール**で再確認すること。  
- 以下に **EC2 と代表的な EBS（gp3）**のみ示す。**データ転送**（NAT・CloudFront・越境など）、**Elastic IP の未使用課金**、**バックアップスナップショット**、**Route 53** 等は含めない。

### 8-1. EC2（On-Demand、us-west-2 の参考値）

| 区分 | 例（タイプ） | 目安（USD / 時間） | 目安（USD / 月 ※） |
|------|----------------|-------------------|---------------------|
| 最小帯 | **m6i.xlarge**（4 vCPU / 16 GiB） | 約 **0.19** | 約 **140** |
| 推奨帯 | **m6i.2xlarge**（8 vCPU / 32 GiB） | 約 **0.38** | 約 **280** |

※ 月額は **24h × 31 日 ≒ 744 時間** の単純積算（約 **730 時間/月** で **約 140 / 280 USD** 程度とする説明も多い）。**常時起動**のざっくり比較用。

### 8-2. EBS（gp3、us-west-2）

- **ストレージ**: リージョンにより **GB・月あたりおおよそ 0.08 USD 前後**（無料枠・プロビジョンド IOPS 超過は別途）。  
- 例: **100 GB** の gp3 を 1 本 → **月 約 8 USD 前後**のオーダー（詳細は AWS 最新単価表を参照）。

### 8-3. まとめ（オレゴン・単一 EC2 のイメージ）

| 構成イメージ | EC2（月目安） | EBS 100GB（月目安） | 合算（税別・ざっくり） |
|--------------|---------------|---------------------|-------------------------|
| 最小帯（m6i.xlarge） | 約 140 USD | 約 8 USD | **約 150 USD / 月** |
| 推奨帯（m6i.2xlarge） | 約 280 USD | 約 8 USD | **約 290 USD / 月** |

**Reserved Instance / Savings Plans** を適用すると、上記オンデマンド比で **割安**になる場合が多い。検証環境は **夜間停止**などで EC2 時間を減らすと費用を抑えやすい。
