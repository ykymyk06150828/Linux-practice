# 本番環境構築手順書

**対象読者**: 本アプリを初めて触る運用担当者・インフラ担当者  
**目的**: **1 台のサーバ（例: AWS EC2）** に、リポジトリの構成どおり **PostgreSQL / Redis / Nginx（Docker）** と **バックエンド（Node）**・**フロント（Next.js）** を載せ、ブラウザから HTTPS で利用できる状態にする。  

**前提**: サーバ OS は **Amazon Linux 2023**（AWS EC2 の公式 AMI など）。**root または sudo 可能なユーザー**（多くの場合 `ec2-user`）で作業できること。  
**スペック目安**: `docs/11_AWS_EC2構築スペック案.md` を参照。

---

## このアプリの構成（先に読むと迷いにくい）

| 名前 | 役割 | ポート（既定） |
|------|------|----------------|
| **Nginx** | ユーザーがアクセスする入口。**HTTPS** と **パス振り分け** | 80 / 443 |
| **Next.js（web）** | 画面（HTML）。`/api` や `/ws` は中継しない（※） | 3000 |
| **API（server）** | ログイン・データ・**ターミナル用 WebSocket** | 3001 |
| **PostgreSQL** | ユーザー・コース等のデータ | 5432（内部のみ推奨） |
| **Redis** | セッション等 | 6379（内部のみ推奨） |
| **Docker Engine** | 受講者ごとの **Linux 研修コンテナ** を起動 | ソケット `/var/run/docker.sock` |

※ ブラウザは **同じドメイン** でページを開き、`/api/...` へアクセスすると Next 側設定で **裏の API（3001）** に転送されます。ターミナルは **`wss://` で `/ws/`** へ接続します。

**データの流れ（イメージ）**

```
インターネット → [443 HTTPS] → Nginx → 静的/画面は Next:3000
                              → /api/, /ws/ は API:3001
```

## 作業の全体流れ

1. サーバを用意し、**セキュリティグループ（ファイアウォール）** を設定する  
2. **Docker**・**Node.js 20**・**Git** を入れる  
3. **ソースコード** をサーバに置く  
4. **研修用 Docker イメージ** をビルドする  
5. **PostgreSQL / Redis / Nginx** を `infra/` の Compose で起動する  
6. **データベースマイグレーション** と（初回のみ）初期データの検討  
7. **バックエンド** を本番モードで起動する  
8. **フロント** を本番ビルドして起動する  
9. **TLS 証明書** を設定し、**環境変数（公開 URL・CORS）** を本番用に直す  
10. **動作確認** と **初期パスワード変更**

順番を変えると 502 エラーや接続エラーになりやすいので、上から順に進めてください。

## 1. サーバの準備

### 1-1. マシンスペック

- `docs/11_AWS_EC2構築スペック案.md` の **推奨帯**（例: 8 vCPU / 32 GB メモリ）を目安にする。  
- **EBS**: gp3、十分な容量（ルート 50 GB 以上推奨）。

### 1-2. 開放するポート（セキュリティグループ）

| 方向 | ポート | 誰から | 用途 |
|------|--------|--------|------|
| インバウンド | **22** | **管理用 IP のみ** | SSH |
| インバウンド | **80** | 任意（最終的に HTTPS へリダイレクトするなら全世界でも可） | HTTP |
| インバウンド | **443** | 利用者（研修生・管理者） | HTTPS |

**PostgreSQL(5432)・Redis(6379) をインターネットに開けないこと。**（`docs/05_インフラ構成書.md` と同じ考え方）

### 1-3. 固定パブリック IP（任意）

- ドメインを当てる場合は **Elastic IP** などで IP を固定すると、DNS が楽です。

## 2. OS に必要なソフトを入れる（Amazon Linux 2023）

入れるものの役割は次のとおりです。

| ソフト | 用途 |
|--------|------|
| **Git** | ソース取得 |
| **Docker Engine** + **Compose プラグイン**（`docker compose`） | `infra` の PostgreSQL / Redis / Nginx |
| **Node.js 20 以上** | `server` / `web` のビルド・実行 |

以下は **Amazon Linux 2023** 向けです。コマンドは **管理者権限（sudo）** が必要です。

### 2-1. パッケージのインストール

```bash
sudo dnf update -y
sudo dnf install -y git

# Docker Engine（リポジトリにあれば Compose プラグインも同時に）
sudo dnf install -y docker docker-compose-plugin || sudo dnf install -y docker
sudo systemctl enable --now docker
sudo usermod -aG docker ec2-user
```

**`ec2-user`** は、Amazon Linux 2023 の EC2 で **SSH ログインに使うデフォルトユーザー**です。`docker` グループにこれを入れると、**`sudo` なしで `docker` / `docker compose` を実行**できます。別名のユーザーで運用する場合は、`usermod` の引数をそのユーザー名に読み替えてください。

`docker` グループを付けたあとは **一度ログアウトして SSH し直す**か、`newgrp docker` でグループを反映してから `docker` / `docker compose` を実行してください。

**Docker Compose V2**（上記で `docker compose version` が失敗する場合の例。`x86_64` / `aarch64` 向けバイナリが取得されます）

```bash
sudo mkdir -p /usr/libexec/docker/cli-plugins
sudo curl -fsSL "https://github.com/docker/compose/releases/latest/download/docker-compose-linux-$(uname -m)" \
  -o /usr/libexec/docker/cli-plugins/docker-compose
sudo chmod +x /usr/libexec/docker/cli-plugins/docker-compose
```

AL2023 に付属する `docker` パッケージの扱いは [Docker のパッケージ（Amazon Linux 2023 User Guide）](https://docs.aws.amazon.com/linux/al2023/ug/docker.html) を参照してください。

**Node.js 20（NodeSource の RPM リポジトリ）**

```bash
curl -fsSL https://rpm.nodesource.com/setup_20.x | sudo bash -
sudo dnf install -y nodejs
```

### 2-2. インストール確認

```bash
git --version
docker --version
docker compose version
node -v    # v20.x 以上
npm -v
```

上記で **Node が v20 未満**のときは、[Node.js 公式](https://nodejs.org/) の **LTS（20.x）** 手順に従って入れ直してください。**nvm** を使う場合は、デプロイ用ユーザーで `nvm install 20` → `nvm use 20` し、**systemd の `ExecStart` からも同じ `node` が参照できる**ようにしてください（本手順の §8 では `/usr/bin/node` を想定しています）。

## 3. デプロイ用ユーザとディレクトリ

本手順では、例として **`/opt/linuxtrainer`** にアプリを置く前提で書きます（別パスでも可）。

```bash
sudo mkdir -p /opt/linuxtrainer
sudo chown ec2-user:ec2-user /opt/linuxtrainer
cd /opt/linuxtrainer
```

所有者の **`ec2-user`** は §2 の Docker グループ追加と同じく、EC2 のデフォルトユーザーを想定しています。別ユーザーでデプロイする場合は `chown` を読み替えてください。

ソースは **Git クローン** か **ZIP 展開** のどちらかで配置します。

### GitHub から `git clone` するときの認証

GitHub は **Git 操作の HTTPS 認証に、アカウントのログインパスワードを受け付けません**。`Password authentication is not supported` や `Invalid username or token` となるのはこのためです。次の **いずれか** を用意してください。

| 方法 | 手順の要点 |
|------|------------|
| **HTTPS + Personal Access Token（PAT）** | [GitHub → Settings → Developer settings → Personal access tokens](https://github.com/settings/tokens) でトークンを発行する。**Classic** の場合は `repo` スコープ（private リポジトリの場合）。**Fine-grained** の場合は対象リポジトリへの **Contents: Read-only** など。`git clone` で `Username` を聞かれたら **GitHub ユーザー名**、`Password` を聞かれたら **パスワードではなくトークン文字列** を貼り付ける。 |
| **SSH**（よく使われる） | EC2 上で `ssh-keygen -t ed25519 -C "deploy@linuxtrainer" -f ~/.ssh/github_linuxtrainer -N ""` などで鍵を作成し、**公開鍵**（`.pub`）を GitHub の **Settings → SSH and GPG keys** に登録。クローン URL は **HTTPS ではなく SSH** を使う（下記 `git@github.com:...`）。初回は `Are you sure you want to continue connecting` で `yes`。 |

**セキュリティ**: PAT や秘密鍵は **リポジトリにコミットしない**こと。サーバにファイルで残す場合はパーミッションを絞る（例: `chmod 600 ~/.ssh/github_linuxtrainer`）。

**空のディレクトリ**（上例の `/opt/linuxtrainer`）に直接展開する場合（HTTPS の例）:

```bash
git clone https://github.com/masayakurosawa-git/linux-master-app.git .
```

同じく **SSH** で取る場合:

```bash
git clone git@github.com:masayakurosawa-git/linux-master-app.git .
```

**サブディレクトリ名**で取りたい場合（例: ホーム直下など）:

```bash
git clone https://github.com/masayakurosawa-git/linux-master-app.git
cd linux-master-app
```

SSH に読み替える場合は、1 行目を `git clone git@github.com:masayakurosawa-git/linux-master-app.git` にする。

以降の手順は、リポジトリの **ルート**（`infra/`・`server/`・`web/` がある場所）をカレントディレクトリにして実行してください（2 つ目のパターンでは `cd linux-master-app` した先がルートです）。ZIP や別経路で配布物を置く場合も、**同じディレクトリ構成**になるように展開してください。

## 4. 研修用 Linux コンテナのイメージをビルド

受講者のターミナルは、このイメージからコンテナが作られます。**API を起動する前に** 1 回必要です。

リポジトリルートで:

```bash
docker build -t linuxtrainer-learner:latest -f server/docker/learner/Dockerfile server/docker/learner
```

- **`-f` の後** … Dockerfile のパス  
- **最後の引数** … **ビルドコンテキスト**（コピー元のディレクトリ）。**必須**で、`-f` と同じパスでも **別引数として付ける**必要があります。  
  - 付け忘れると `docker buildx build requires exactly 1 argument` のようになります。  
  - `server/docker/learner` だけをシェルに打っても（ディレクトリへの `cd` ではなく）コマンドにはならないので、**上記 1 行をそのまま**実行してください。

成功すると `docker images` に `linuxtrainer-learner` が表示されます。

### よくあるエラー（Docker）

| メッセージ | 原因と対処 |
|------------|------------|
| `requires exactly 1 argument` / `PATH \| URL \| -` | **ビルドコンテキスト**（行末の `server/docker/learner`）が欠けている。上記コマンドを **1 行すべて**実行する。 |
| `permission denied` … `docker.sock` | `ec2-user` に **`docker` グループがまだ効いていない**（§2-1 の `usermod` の直後はよくある）。**SSH を一度切ってログインし直す**か、同じセッションで `newgrp docker` を実行してから `docker build` を再試行する。暫定的に **`sudo docker build ...`**（同じ引数）でもビルドできる。 |

## 7. フロント（web）のビルド用環境変数

```bash
cd /opt/linuxtrainer/web
cp .env.example .env.local
```

**本番で重要な値:**

| 変数 | 例 | 説明 |
|------|-----|------|
| `BACKEND_URL` | `http://127.0.0.1:3001` | **同一サーバ内**の API へ Next が中継する先。通常はこのまま |
| `NEXT_PUBLIC_WS_URL` | `wss://研修.example.com/ws/terminal` | **ブラウザから接続する WebSocket の公開 URL**（パスは `/ws/terminal` 固定）。サイトが **`https://`** なら **`wss://`**、**`http://`** だけなら **`ws://`**。ホスト名・IP は **ユーザーがアドレスバーに入力するものと同じ**にする |

**パブリック IP でアクセスする例**（Nginx が 443 で TLS、HTTP は 80 の一般的な構成）:

| ユーザーが開く画面の URL | `NEXT_PUBLIC_WS_URL` に設定する値 |
|---------------------------|-------------------------------------|
| `https://34.222.207.38/...` | `wss://34.222.207.38/ws/terminal` |
| `http://34.222.207.38/...`（TLS なし） | `ws://34.222.207.38/ws/terminal` |

- **ポート**は、ブラウザで **443 / 80 を使う通常の HTTPS・HTTP** なら URL に **書かない**（上表のとおり）。  
- 例: `https://34.222.207.38:8443` で開く構成なら `wss://34.222.207.38:8443/ws/terminal` のように **ホストと同じポート**を付ける。

**変更手順**: `web/.env.local` を編集して上記の 1 行を入れる（または既存の `NEXT_PUBLIC_WS_URL` を書き換える）→ **`npm run build` をやり直す** → Next のプロセスを再起動（§8-2 の `systemctl restart` など）。

※ `NEXT_PUBLIC_*` は **ビルド時**に埋め込まれるため、**値を変えたら必ず `npm run build` し直し**が必要です。

```bash
npm ci
npm run build
```

## 8. 本番としてプロセスを動かす（systemd 例）

開発用の `npm run dev` は使わず、**ビルド済み**を常駐させます。

### 8-1. API

```bash
cd /opt/linuxtrainer/server
npm run build
```

実行コマンドは **`node dist/main.js`**（`package.json` の `start` と同じ）。

**systemd ユニット例** `/etc/systemd/system/linuxtrainer-api.service`:

```ini
[Unit]
Description=Linuxtrainer API
After=network.target docker.service

[Service]
Type=simple
User=ec2-user
WorkingDirectory=/opt/linuxtrainer/server
EnvironmentFile=/opt/linuxtrainer/server/.env
ExecStart=/usr/bin/node dist/main.js
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

**`User=ec2-user`** は §2 で `docker` グループに入れたユーザーと揃えています。`docker.sock` 経由で研修コンテナを起動できる必要があります。別ユーザーで動かす場合はここも読み替え、そのユーザーに `docker` グループを付与してください。

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now linuxtrainer-api.service
sudo systemctl status linuxtrainer-api.service
```

確認:

```bash
curl -sS http://127.0.0.1:3001/api/health
curl -sS http://127.0.0.1:3001/api/ready
```

`ready` が JSON で `database` / `redis` が ok であること。

### 8-2. Web（Next.js）

```bash
cd /opt/linuxtrainer/web
npm run start
```

常駐用 **systemd 例** `/etc/systemd/system/linuxtrainer-web.service`:

```ini
[Unit]
Description=Linuxtrainer Next.js
After=network.target linuxtrainer-api.service

[Service]
Type=simple
User=ec2-user
WorkingDirectory=/opt/linuxtrainer/web
EnvironmentFile=/opt/linuxtrainer/web/.env.local
ExecStart=/usr/bin/npm run start --prefix /opt/linuxtrainer/web
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

または `npx next start -p 3000` を `ExecStart` に直接書いてもよいです。

```bash
sudo systemctl enable --now linuxtrainer-web.service
```

ブラウザ用に **3000** で待ち受けていることを確認します。

```bash
curl -sS -o /dev/null -w "%{http_code}" http://127.0.0.1:3000/
```

**`npm run start` で `ReferenceError: self is not defined` が出る場合**（`/learner/terminal` 関連のスタックトレース）:

- **原因**: ターミナル UI（xterm.js）が **ブラウザ専用の `self`** を参照するのに、Next の **サーバー側バンドル**に取り込まれていると、Node 上で実行されたときに失敗します。  
- **対処**: リポジトリの **最新**を取得し、`web` で **`npm run build` をやり直してから** `npm run start`（または systemd 再起動）してください。本リポジトリでは `TerminalPane` を **`next/dynamic` の `ssr: false`** で読み込み、本番サーバー上では xterm をバンドルしないようにしています。

## 9. Nginx の前段と HTTPS

`infra` の Nginx は **ホストの `host.docker.internal:3000` と `:3001`** にプロキシします（Linux では `extra_hosts: host-gateway` 済み）。

1. **証明書**  
   - **Let's Encrypt（certbot）** で `nginx/certs/` に配置する、または  
   - **ACM + ALB** で TLS を終端し、その背後に Nginx / アプリを置く（構成は組織の標準に合わせる）

2. **HTTPS 用設定**  
   - `infra/nginx/conf.d/https.conf.example` を `https.conf` にコピーし、**証明書パス**を本番用に修正  
   - `docker-compose.yml` の Nginx で **443 を公開**する行のコメントを外す  

3. **コンテナ再起動**

```bash
cd /opt/linuxtrainer/infra
docker compose up -d --force-recreate nginx
```

4. **`FRONTEND_ORIGIN`**（server）と **`NEXT_PUBLIC_WS_URL`**（web）が **実際の `https://` の URL** と一致しているか再確認。変更したら **API 再起動** と **web の `npm run build` し直し**。

## 11. 本番で必ず行うこと（セキュリティ）

- **初期パスワード**（seed を使った場合）を **直ちに変更**する  
- **`.env` の秘密情報**を Git にコミットしない  
- **SSH は鍵認証**、**22 番は限定 IP**  
- **バックアップ**（EBS スナップショットや PostgreSQL のダンプ）方針を決める（`docs/05_インフラ構成書.md` 8 章）

## 12. 更新・再デプロイの流れ

1. アプリを止める（順不同だが、先に Web を止めるとユーザーへの影響が少ない）  
   `sudo systemctl stop linuxtrainer-web` / `linuxtrainer-api`  
2. `git pull` などでソース更新  
3. `server`: `npm ci` → `npm run build` → `npx prisma migrate deploy` → `sudo systemctl restart linuxtrainer-api`  
4. `web`: `npm ci` → `npm run build` → `sudo systemctl restart linuxtrainer-web`  
5. 研修イメージの Dockerfile を変えた場合は **イメージの再ビルド**
