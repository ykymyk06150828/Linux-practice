# Linux コマンド研修アプリ — 利用手順

本書は、現時点でリポジトリに含まれる **インフラ（PostgreSQL / Redis / Nginx）** の起動・確認方法をまとめたものです。アプリケーション本体の起動はフェーズ6以降で追加される想定です。

---

## 1. 前提条件

次のソフトウェアがインストールされ、起動できる状態であること。

- **Docker Engine** 24 以降
- **Docker Compose** V2（`docker compose` コマンド）

macOS / Windows では **Docker Desktop**、Linux では OS 標準の手順で Docker を導入してください。

Nginx がホスト上のアプリ（ポート 3000 / 3001）へプロキシする設定のため、**`host.docker.internal`** が名前解決できることが望ましいです。本リポジトリの `infra/docker-compose.yml` では Linux 向けに `extra_hosts: host-gateway` を指定しています。

---

## 2. 配置場所

インフラ定義はリポジトリ直下の **`infra/`** にあります。

```
infra/
  docker-compose.yml    # PostgreSQL, Redis, Nginx
  env.example           # 環境変数のひな形
  nginx/                # Nginx 設定
  scripts/              # 証明書生成・疎通確認
  docker/daemon.json.example  # Docker デーモン設定の参考
```

---

## 3. 初回セットアップ（コマンド手順）

ターミナルで次を実行します。

### 3-1. 作業ディレクトリへ移動

```bash
cd infra
```

### 3-2. 環境変数ファイルの作成

```bash
cp env.example .env
```

`.env` を編集し、少なくとも **`POSTGRES_PASSWORD`** を推測されにくい値に変更してください。  
`DATABASE_URL` / `REDIS_URL` は Docker Compose 内のサービス名（`postgres` / `redis`）をホストにした URL になっている想定です。

### 3-3. スクリプトの実行権限（初回のみ）

```bash
chmod +x scripts/gen-selfsigned-certs.sh scripts/verify-infra.sh
```

### 3-4. TLS 用の自己署名証明書（HTTPS を有効にする場合）

開発環境で Nginx の **HTTPS（443）** を使う場合に実行します。HTTP のみの場合は省略して構いません。

```bash
./scripts/gen-selfsigned-certs.sh
```

証明書は `infra/nginx/certs/` に `server.crt` / `server.key` として生成されます。秘密鍵はリポジトリにコミットしないでください（`.gitignore` 済み）。

HTTPS を有効にする具体的な手順は **「5. HTTPS（TLS）を有効にする場合」** を参照してください。

### 3-5. コンテナの起動

```bash
docker compose up -d
```

### 3-6. 疎通確認

```bash
./scripts/verify-infra.sh
```

期待どおりであれば、PostgreSQL / Redis / Nginx の状態確認と、`http://127.0.0.1/healthz` への疎通が行われます。

---

## 4. 起動するサービスとポート

| サービス | コンテナ内 | ホスト（既定） | 用途 |
|----------|------------|----------------|------|
| PostgreSQL 15 | 5432 | 127.0.0.1:5432 | アプリ用データベース |
| Redis 7 | 6379 | 127.0.0.1:6379 | セッション等 |
| Nginx | 80（および任意で 443） | 0.0.0.0:80 | リバースプロキシ |

Nginx の振り分け（アプリ実装後の想定）:

- **`/`** … フロントエンド（ホストの **3000** 番ポート）
- **`/api/`** … API（ホストの **3001** 番ポート）
- **`/ws/`** … WebSocket（ホストの **3001** 番ポート）

アプリが未起動のとき、これらのパスは **502** などになることがあります。まず **`curl http://127.0.0.1/healthz`** が `ok` を返すか確認してください。

---

## 5. HTTPS（TLS）を有効にする場合

1. 自己署名証明書を **`./scripts/gen-selfsigned-certs.sh`** で生成済みであること。
2. `infra/nginx/conf.d/https.conf.example` を **`https.conf`** にコピーする。  
   ```bash
   cp nginx/conf.d/https.conf.example nginx/conf.d/https.conf
   ```
3. `infra/docker-compose.yml` の **nginx サービス**で、**443 ポート**の行のコメントを外す。
4. コンテナを再起動する。  
   ```bash
   docker compose up -d --force-recreate nginx
   ```

本番環境では、Let's Encrypt、AWS ACM、社内 CA など適切な証明書に差し替えてください。方針は `docs/05_インフラ構成書.md` に記載しています。

---

## 6. 停止・削除

### 6-1. 停止（コンテナを止める）

```bash
cd infra
docker compose stop
```

### 6-2. コンテナ削除（ボリュームは残る）

```bash
docker compose down
```

### 6-3. データボリュームも含めて削除する場合

**PostgreSQL のデータが消えます。** 実行前にバックアップを確認してください。

```bash
docker compose down -v
```

## 7. トラブルシューティング

| 現象 | 確認すること |
|------|----------------|
| `verify-infra.sh` で PostgreSQL に失敗する | `.env` のユーザー名・DB 名・パスワードが `docker-compose.yml` と一致しているか |
| Redis に接続できない | `docker compose ps` で redis が Up か |
| `curl http://127.0.0.1/healthz` が失敗する | Nginx が起動しているか、ポート 80 が他プロセスに占有されていないか |
| ブラウザで 502 | ホストでフロント（3000）・API（3001）が起動しているか（フェーズ6以降） |
| Linux で `host.docker.internal` が効かない | `docker-compose.yml` の `extra_hosts` を環境に合わせて修正する |

## 8. バックエンド API（フェーズ6）

リポジトリの **`server/`** に Node.js（Fastify）+ Prisma + Redis + Docker の API を配置しています。

### 8-1. 前提

- `infra` で PostgreSQL / Redis を起動済みであること（または同等の接続先）
- Docker が動作し、`linuxtrainer-learner` イメージをビルド済みであること

```bash
docker build -t linuxtrainer-learner:latest -f server/docker/learner/Dockerfile server/docker/learner
```

### 8-2. 環境変数

`server/.env.example` を `server/.env` にコピーし、`DATABASE_URL` / `REDIS_URL` を編集する。  
Docker Compose の PostgreSQL に接続する例:

```env
DATABASE_URL=postgresql://app:<パスワード>@127.0.0.1:5432/linuxtrainer
REDIS_URL=redis://127.0.0.1:6379
```

### 8-3. DB とシード

```bash
cd server
npm install
npx prisma generate
npx prisma db push
npm run db:seed
```

### 8-4. 起動

```bash
cd server
npm run dev
```

- API: `http://127.0.0.1:3001/api/...`
- ヘルス: `GET http://127.0.0.1:3001/api/health`
- レディネス（DB / Redis）: `GET http://127.0.0.1:3001/api/ready`（失敗時は 503）
- WebSocket: `ws://127.0.0.1:3001/ws/terminal`（受講者セッション Cookie 必須）

詳細は **`server/README.md`** を参照。

## 9. フロントエンド（フェーズ7）

リポジトリの **`web/`** に Next.js（App Router）の UI を配置しています。

### 9-1. セットアップ

```bash
cd web
cp .env.example .env.local
npm install
npm run dev
```

ブラウザで **http://127.0.0.1:3000** を開く。トップページから受講者／管理者ログインや共通画面（エラー・権限・メンテナンス等）へ遷移してレイアウトを確認できる。

### 9-2. API との連携

- `next.config.ts` が **`/api/*`** をバックエンド（既定 `http://127.0.0.1:3001`）へ **rewrite** する。フロントからは **`fetch("/api/...", { credentials: "include" })`** で呼び出す。
- バックエンドの **`FRONTEND_ORIGIN=http://localhost:3000`**（または `http://127.0.0.1:3000`）を設定し、CORS でフロントのオリジンを許可する。
- WebSocket は **`NEXT_PUBLIC_WS_URL`**（既定 `ws://127.0.0.1:3001/ws/terminal`）へ接続する。Cookie が付与されない場合は、同一ホストの Nginx 配下で公開するか、バックエンドのトークン方式を検討する。

### 9-3. 画面だけ確認する場合

バックエンドを起動しなくても、**`/`**・**`/login`**・**`/admin/login`**・共通ルート（`/error` 等）は表示できる。データ取得が必要なダッシュボードやターミナルは、API・Docker が利用可能な状態で確認する。

詳細は **`web/README.md`** を参照。
