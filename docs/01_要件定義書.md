# 30人同時接続を前提にした Linux コマンド研修アプリ 要件定義

## 1. 概要

### 1-1. システム名
Linux コマンド研修アプリ

### 1-2. 目的
インフラエンジニア研修において、受講者がブラウザ上から Linux コマンドをいつでも練習できる環境を提供する。  
受講者ごとに分離された実行環境を用意し、同時に約30人が接続しても安定して利用できることを目的とする。

### 1-3. 背景
通常、Linux コマンド研修では各受講者に個別のサーバ環境を用意する必要があり、環境構築・管理・初期化の負荷が高い。  
また、Web アプリにターミナル入力画面を一から実装するのは難易度が高いため、ブラウザ埋め込み型のターミナル UI とコンテナ実行環境を組み合わせて、運用しやすく安全な研修基盤を構築する。

---

## 2. システム化の目的

- 受講者がブラウザだけで Linux コマンドを学習できるようにする
- 研修管理者が受講者ごとの環境を簡単に準備・初期化できるようにする
- 同時30人程度の利用に耐えられる構成とする
- 受講者同士の操作影響を防ぐため、実行環境をユーザー単位で分離する
- 危険な操作や過剰なリソース消費を制限し、安全に運用できるようにする

---

## 3. 想定利用者

### 3-1. 受講者
- Linux コマンドを学習する研修受講者
- ブラウザ上でコマンドを入力し、実行結果を確認する

### 3-2. 管理者
- 研修を運営する講師、管理者
- 受講者の利用状況確認
- 研修環境の初期化
- 問題ファイルの配布
- 利用制限や停止処理の実施

---

## 4. 前提条件

- 同時接続数は最大30人程度を想定する
- 利用者は社内研修受講者を想定する
- 受講者は PC のブラウザからアクセスする
- 研修用 Linux 環境は受講者ごとに個別コンテナで提供する
- 実行環境は Docker コンテナ上の Linux シェルとする
- ターミナル画面はブラウザ埋め込み型の UI ライブラリを利用する
- 本番公開用の一般向けサービスではなく、社内研修用システムとして構築する

---

## 5. 対象範囲

### 5-1. 対象
- ログイン機能
- 受講者別ターミナル画面
- Linux コマンド実行機能
- 実行結果表示機能
- 受講者ごとのコンテナ環境払い出し
- 研修ファイルの初期配置
- 環境リセット機能
- 管理者向け受講状況確認機能

### 5-2. 対象外
- 本格的な Linux サーバ構築演習
- root 権限を必要とする学習
- 外部ネットワークへの自由な接続
- 本システム外の本番サーバへの接続
- 受講者同士の共同編集機能

---

## 6. システム構成方針

## 6-1. 全体構成
本システムは以下の構成を前提とする。

- フロントエンド: Next.js
- ターミナル UI: xterm.js
- バックエンド API: Node.js
- 通信方式: WebSocket
- 実行環境: Docker コンテナ
- セッション管理: Redis
- リバースプロキシ: Nginx
- インフラ基盤: AWS または同等のクラウド環境

## 6-2. 基本構成イメージ
- 受講者はブラウザで Web アプリにアクセスする
- Web アプリ内に埋め込まれたターミナル画面でコマンドを入力する
- 入力内容は WebSocket 経由でバックエンドへ送信する
- バックエンドは受講者ごとの Docker コンテナ内シェルへ接続する
- 実行結果をリアルタイムでブラウザへ返却する
- 各受講者は他の受講者と分離されたコンテナ環境を利用する

---

## 7. 機能要件

## 7-1. 認証機能
### 要件
- 受講者はログイン後に利用できること
- 管理者は管理者権限でログインできること
- ログイン後はセッション管理を行うこと

### 詳細
- ID/パスワード認証を基本とする
- 将来的に SSO 連携を拡張可能な構成とする
- 未認証状態ではターミナル画面へ遷移できないこと

---

## 7-2. ターミナル表示機能
### 要件
- ブラウザ内に Linux ターミナル画面を表示できること
- 受講者がコマンド入力、Enter 実行、スクロールを行えること
- 実行結果がリアルタイムで反映されること

### 詳細
- xterm.js 等のブラウザ埋め込み型ターミナル UI を利用する
- フォント、背景色、カーソル表示など、最低限の視認性を確保する
- 画面サイズ変更時にターミナルが追従すること

---

## 7-3. コマンド実行機能
### 要件
- 受講者は自分専用の Linux 環境でコマンドを実行できること
- 実行結果を画面上で確認できること
- 基本的な Linux 学習コマンドが使えること

### 想定コマンド例
- pwd
- ls
- cd
- cat
- cp
- mv
- rm
- mkdir
- touch
- grep
- find
- chmod
- chown
- ps
- top
- df
- du
- head
- tail
- less
- echo
- history

### 補足
- root 権限が必要な操作は禁止する
- sudo は原則使用不可とする
- 危険コマンドは制限対象とする

---

## 7-4. 受講者別環境払い出し機能
### 要件
- 受講者ごとに個別の Docker コンテナを払い出すこと
- 他受講者の操作が影響しないこと
- 同時30人程度の接続でも利用できること

### 詳細
- 1ユーザー1コンテナ方式を採用する
- ログイン時に利用中コンテナを割り当てる
- 未作成の場合は新規作成する
- 利用終了時は停止または破棄できること

---

## 7-5. 初期ファイル配布機能
### 要件
- 練習用ファイルやディレクトリを受講者環境へ事前配置できること
- 問題ごとに初期状態を再現できること

### 配置対象例
- テキストファイル
- ログファイル
- ディレクトリ構造
- 権限設定済みファイル
- 課題用シェルスクリプト

---

## 7-6. 環境リセット機能
### 要件
- 受講者環境を初期状態に戻せること
- 管理者は任意受講者の環境をリセットできること
- 受講者自身でも必要に応じてリセット申請または操作できること

### 詳細
- コンテナ再生成方式を基本とする
- 初期配布ファイルを再配置する
- 進捗情報を保持する場合は別途 DB 保存する

---

## 7-7. 実行履歴保存機能
### 要件
- 受講者が実行したコマンド履歴を保存できること
- 管理者が履歴を確認できること

### 保存対象
- 実行日時
- ユーザーID
- 実行コマンド
- 実行結果ステータス
- 接続開始/終了時刻

### 利用目的
- 学習進捗確認
- 操作ログ確認
- トラブル調査

---

## 7-8. 管理機能
### 要件
- 管理者が受講者一覧を確認できること
- 接続中ユーザー数を確認できること
- 受講者ごとの環境状態を確認できること
- 必要に応じて環境停止、削除、リセットができること

### 管理画面で確認したい項目
- ユーザー名
- 接続状態
- 最終アクセス時刻
- 割り当てコンテナ名
- CPU使用率
- メモリ使用量
- 実行中/停止中状態

---

## 7-9. 接続制御機能
### 要件
- 一定時間無操作の受講者は自動切断または自動停止できること
- 同一アカウントでの多重ログイン制御ができること

### 詳細
- 無操作 15〜30 分で警告表示
- 一定時間後にコンテナ停止候補とする
- 同一ユーザーの複数接続は制限または上書き制御を行う

---

## 8. 非機能要件

## 8-1. 性能要件
- 同時接続数30人で安定動作すること
- ターミナル入力から画面反映までの遅延は通常 1 秒以内を目標とする
- ログイン後、受講者環境への接続開始まで 10 秒以内を目標とする
- 事前起動済みコンテナがある場合は 3 秒以内を目標とする

---

## 8-2. 可用性要件
- 研修時間中に安定動作すること
- 単一障害点を把握した上で、初期構成ではシンプルな単一構成を採用する
- 将来的に複数台構成へ拡張可能な設計とする

---

## 8-3. 拡張性要件
- 将来的に 50人以上への拡張が可能なこと
- 研修コースごとに初期ファイルセットを切り替えられること
- 管理画面や採点機能を追加できる構成とする
- コンテナホストを複数台に増やせること

---

## 8-4. セキュリティ要件
- 受講者環境は他ユーザーから分離されていること
- コンテナは非特権モードで実行すること
- root 権限の利用は禁止または厳格に制御すること
- ホスト OS に直接アクセスできないこと
- 外部ネットワーク接続は必要最小限に制限すること
- 危険コマンドや過剰負荷コマンドに対して制限方針を設けること
- 通信は HTTPS 化すること

---

## 8-5. 運用要件
- 管理者が受講者環境の起動・停止・再作成を行えること
- 障害時にログを確認できること
- コンテナの CPU / メモリ使用量を監視できること
- 受講者接続数を可視化できること

---

## 9. リソース要件

## 9-1. 同時接続前提
- 最大同時接続数: 30人
- 基本は 1ユーザー1コンテナ
- 余剰を含めて 35〜40 コンテナ程度を扱える構成を想定する

## 9-2. コンテナ制限例
- CPU 上限: 0.25〜0.5 vCPU / コンテナ
- メモリ上限: 128MB〜256MB / コンテナ
- ディスク使用量上限を設定すること
- プロセス数上限を設定すること

## 9-3. 初期サーバ想定
- App / API / WebSocket / Docker ホスト兼用サーバ
- 4 vCPU 以上
- 8GB メモリ 以上
- SSD ストレージ利用
- Redis 用リソースを確保すること

※ 実際の必要スペックは、利用コマンド範囲や演習内容に応じて調整する

---

## 10. 制約条件

- ブラウザのみで利用可能であること
- 専用クライアントソフトのインストールを不要とすること
- 研修用のため、操作自由度より安全性を優先すること
- 本番 Linux サーバへの接続機能は持たせないこと
- 受講者がホスト環境に影響を与えられないこと

## 17. 受入条件

### 必須受入条件
- 30人同時接続で利用可能であること
- 受講者ごとに独立した環境が割り当てられること
- ターミナル画面で Linux コマンドを実行できること
- 実行結果がリアルタイムに表示されること
- 管理者が受講者環境を確認できること
- 環境リセットが正常に行えること
- 危険な操作が一定範囲で制限されていること

---

## 18. 想定リスク

- 受講者の過剰なコマンド実行による高負荷
- WebSocket 切断によるターミナル不安定化
- コンテナ起動遅延による利用開始待ち
- 危険コマンド実行によるリソース圧迫
- 単一サーバ障害時の全体停止

### 対策方針
- コンテナごとに CPU / メモリ上限設定
- 無操作タイムアウト制御
- 事前起動コンテナプール導入
- 危険コマンド制限
- 監視と即時復旧手順整備

---

## 19. まとめ

本システムは、Linux コマンド研修をブラウザ上で安全かつ効率的に実施するための研修基盤である。  
同時30人程度の接続を想定し、受講者ごとに分離されたコンテナ環境を提供することで、安全性・拡張性・運用性を両立する。  
初期構成はシンプルな単一サーバ構成としつつ、将来的にはスケールアウト可能な構成へ拡張できる設計とする。
